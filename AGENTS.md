# AGENTS.md

## Project Type

Hexlet educational project — "Book a call" scheduling app (Cal.com-like, no auth).

## Agent Task

Implement a minimal meeting booking service driven by the TypeSpec API contract in `packages/api/`.

### Domain Entities

| Entity | Description |
|--------|-------------|
| **Owner** | Single predefined calendar owner profile. Used by the admin UI by default. Not managed via API (no auth). |
| **EventType** | Bookable meeting type with `id`, `name`, `description`, `durationMinutes`. Owner creates/updates/deletes; guests read. |
| **Slot** | Computed available time window (`startsAt`, `endsAt`). Not persisted. Generated from event type duration and existing bookings within the 14-day window. |
| **Booking** | Confirmed reservation with guest contact info. One booking per `startsAt` across all event types. |

### Business Rules

1. **No authentication** — owner is a hardcoded profile; guests book without accounts.
2. **14-day booking window** — slots and bookings are only allowed from today through 14 days ahead (inclusive).
3. **No double booking** — the same `startsAt` cannot be booked twice, even for different event types (HTTP 409).
4. **Design-first** — `packages/api/src/main.tsp` is the source of truth. OpenAPI is generated; do not hand-edit `tsp-output/`.

### Guest Scenario

1. Browse event types → `GET /api/event-types`
2. View event type details → `GET /api/event-types/{id}`
3. Pick a date and see free slots → `GET /api/event-types/{id}/slots?date=YYYY-MM-DD`
4. Book a slot → `POST /api/bookings`

### Owner Scenario

1. Manage event types → `POST/PUT/DELETE /api/event-types`
2. View upcoming meetings → `GET /api/bookings` (sorted by `startsAt` ASC)
3. Cancel a meeting → `DELETE /api/bookings/{id}`

## CI/Testing

- CI via `.github/workflows/hexlet-check.yml` is auto-generated by Hexlet — **do not modify**
- Tests run via Hexlet project action: `hexlet-project-action@release`
- E2E tests run via `.github/workflows/e2e.yml` (Playwright, guest booking flow)

## Commit messages

All commits — including those made by agents — must follow [Conventional Commits](https://www.conventionalcommits.org/). See [CONTRIBUTING.md](CONTRIBUTING.md) for the full guide.

Format: `<type>[optional scope]: <description>`

| Type | Use for |
|------|---------|
| `feat` | New functionality (minor version bump) |
| `fix` | Bug fix (patch version bump) |
| `docs` | Documentation only |
| `test` | Tests only |
| `ci` | CI/CD changes |
| `chore` | Maintenance without behavior change |
| `feat!` / `BREAKING CHANGE:` | Breaking changes (major version bump) |

Examples:

- `feat(booking): add slot availability indicator`
- `fix(api): return 409 on double booking`
- `ci: add playwright e2e workflow`

Release changelog and version bumps are generated by release-please from these commit messages.

## Getting Started

```bash
pnpm install
pnpm --filter @app/api build   # compile TypeSpec → OpenAPI
```

Generated output: `packages/api/tsp-output/@typespec/openapi3/openapi.yaml`
